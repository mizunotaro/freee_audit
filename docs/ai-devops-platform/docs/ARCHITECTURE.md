# アーキテクチャ設計書

## 1. システム概要

### 1.1 目的
OpenCode + Z.AI GLM5 + GitHub Actions を用いた**完全自動化開発プラットフォーム**の構築。

### 1.2 コアバリュー
- **自律性**: 人間の介入なしでタスクを完遂
- **安全性**: 品質ゲート・セキュリティチェック自動実行
- **再現性**: 同じ入力 → 同じ出力（temperature=0.0）
- **透明性**: 全ての判断プロセスをログ・PRで可視化

## 2.2 中央ハブ方式の詳細

### 構成要素

| コンポーネント | リポジトリ | 役割 |
|--------------|-----------|------|
| 中央ハブ | ai-devops-platform | タスク収集・実行・PR作成 |
| ターゲット | freee_audit等 | Issue登録・PR受取 |

### データフロー

1. ターゲットリポジトリで `ai:ready` ラベル付きIssue作成
2. 中央ハブが15分ごとにスキャン
3. 優先度順にタスクキューに追加
4. GLM5セッション制限内で順次処理
5. ターゲットリポジトリにPR作成

### セッション制御

| 時間帯 | 並列セッション数 | 理由 |
|--------|-----------------|------|
| 平日昼間 (9:00-18:00) | 2 | 業務利用を優先 |
| それ以外 | 3 | 制限緩和 |

### 利点

- ✅ GLM5セッションのレートリミットを確実に制御
- ✅ 複数リポジトリのタスクを一元管理
- ✅ ターゲットリポジトリへのGitHub Actions設定が不要
- ✅ 優先度ベースのタスクスケジューリング

## 3. コンポーネント詳細

### 3.1 トリガーシステム（ハイブリッド）

| トリガー | タイミング | 用途 |
|---------|-----------|------|
| 定期ポーリング | 15分ごと | バックグラウンドタスク処理 |

### 3.2 タスク選択ロジック

```typescript
interface TaskSelection {
  priority: 'critical' | 'high' | 'medium' | 'low'
  labels: string[]
  age: number  // Issue作成からの経過時間（時間）
  dependencies: number[]  // 依存するIssue番号
}

// 優先度スコア計算
function calculateScore(task: TaskSelection): number {
  const priorityWeight = { critical: 100, high: 75, medium: 50, low: 25 }
  const ageBonus = Math.min(task.age, 48)  // 最大48時間でキャップ
  
  let score = priorityWeight[task.priority] + ageBonus
  
  // 依存関係チェック
  if (task.dependencies.some(dep => !isCompleted(dep))) {
    score = 0  // 依存先が未完了ならスキップ
  }
  
  return score
}
```

## 4. 安定性・信頼性設計

### 4.1 タイムアウト設定

| コンポーネント | タイムアウト | 理由 |
|---------------|-------------|------|
| セッション実行 | 30分 | GLM5セッション上限、長時間実行防止 |
| LLM API呼び出し | 30秒 | API応答待ち上限 |
| GitHub API呼び出し | 60秒 | ネットワーク遅延許容 |
| 品質ゲート（各） | 10分 | ビルド・テスト実行時間 |
| ワークフロー全体 | 1時間 | GitHub Actions制限、強制終了 |

### 4.2 バックオフ戦略

| 試行回数 | 待機時間 | 累計待機 |
|----------|----------|----------|
| 1回目 | 60秒 | 60秒 |
| 2回目 | 120秒 | 180秒 |
| 3回目 | 240秒 | 420秒 |
| 4回目以降 | 300秒（最大） | +300秒 |

### 5.2 例外処理マトリクス

| エラー種別 | 分類 | アクション | 最大試行 | 通知 |
|-----------|------|-----------|----------|------|
| ネットワークタイムアウト | Retryable | 指数バックオフでリトライ | 3回 | なし |
| GitHub API レート制限 | Retryable | 待機後リトライ | 3回 | なし |
| LLM API エラー | Recoverable | フォールバックモデルへ | 1回 | あり |
| セッションハング | Recoverable | 新セッション起動 | 1回 | あり |
| 型エラー | Recoverable | LLM自動修正 | 3回 | あり |
| テスト失敗 | Recoverable | LLM自動修正 | 3回 | あり |
| 認証エラー | Fatal | 人間確認 | 0回 | 即時 |
| 予算超過 | Fatal | 人間確認 | 0回 | 即時 |
| セキュリティ違反 | Fatal | 人間確認 | 0回 | 即時 |

### 5.3 エッジケース対応

| ケース | 検出条件 | 対応アクション |
|--------|----------|---------------|
| 複数ターゲット同時スキャン | 複数リポジトリに同時Issue | 優先度スコア順で逐次処理 |
| 重複Issue検出 | 同一タイトル・同一リポジトリ | 後続をスキップ、コメント通知 |
| セッション上限到達 | 実行中セッション = 上限 | 待機キューに追加、順次処理 |
| PR競合 | 既存PRと競合 | 自動リベース、失敗時はai:blocked |
| 空Issue | 本文が空 | ai:blocked、テンプレート案内 |
| 巨大Issue | 本文 > 10000文字 | ai:blocked、分割推奨コメント |
