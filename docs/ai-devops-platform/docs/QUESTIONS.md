### Q-003: セッション状態の永続化方法

**質問**
セッション状態（進行中のタスク情報）をどこに永続化するか？

**背景**
- GitHub Actions は実行間で状態を保持しない
- セッションが強制終了された場合の復旧が必要

**検討中の選択肢**
1. **GitHub Issue コメント**: メタデータをコメントに保存
2. **GitHub Actions Cache**: キャッシュとして保存（7日制限）
3. **外部ストレージ**: S3, Redis 等（コスト・複雑さ増）
4. **ファイルコミット**: 専用ファイルをリポジトリにコミット

**影響**
- 障害復旧・再開機能の実装

**ステータス**: 🟡 要議論

## 🟡 中優先度（Phase 2で解決）

### Q-004: 大規模リファクタリングの自動化戦略

**質問**
10ファイル以上・数百行の変更が必要なタスクをどう処理するか？

**背景**
- LLM のコンテキストウィンドウには限界がある
- 一度の生成で大規模変更は品質が低下しがち

**検討中の選択肢**
1. **タスク自動分割**: LLM に分割計画を立てさせ、順次実行
2. **段階的リファクタリング**: 複数PRに分割
3. **人間に確認**: 大規模タスクは人間の承認を必須化
4. **制限**: 大規模タスクは自動化対象外とする

**影響**
- 自動化の適用範囲

**ステータス**: 🟡 要議論

### Q-005: 複数Issueの依存関係管理

**質問**
「Issue #123 が完了してから Issue #124 を実行」のような依存関係をどう管理するか？

**背景**
- 機能追加 → テスト作成 → ドキュメント更新 の順序制御が必要
- 依存関係の表現方法が未定

**検討中の選択肢**
1. **ラベル**: `depends-on-123` ラベルで表現
2. **Issue 本文**: 規定フォーマットで依存関係を記述
3. **GitHub Projects**: Projects のボードで依存関係を管理
4. **プロジェクト独自設定**: `config/dependencies.json` で管理

**影響**
- タスクスケジューリングの複雑さ

**ステータス**: 🟡 要設計

### Q-006: LLM コストの予算管理

**質問**
月間LLMコストの上限をどう管理・強制するか？

**背景**
- 無制限に実行するとコストが肥大化
- 予算超過時の動作を決定必要

**検討中の選択肢**
1. **事前チェック**: セッション開始前に月次予算確認
2. **タスク優先度**: 高優先度のみ予算内で実行
3. **アラートのみ**: 超過時に通知、実行は継続
4. **ハードリミット**: 超過時に自動停止

**影響**
- 運用コスト・可用性

**ステータス**: 🟡 要決定

---

### Q-007: 緊急停止メカニズム

**質問**
実行中のセッションを緊急停止する方法は？

**背景**
- バグで無限ループに入った場合
- セキュリティ問題が発覚した場合
- 誤って重要なファイルを変更しようとしている場合

**検討中の選択肢**
1. **`/opencode-stop` コメント**: Issue にコメントで停止
2. **ラベル**: `do-not-process` ラベルで停止
3. **GitHub Actions キャンセル**: ワークフローを手動キャンセル
4. **外部API**: 緊急停止用のエンドポイント

**影響**
- 障害対応速度

**ステータス**: 🟡 要実装
