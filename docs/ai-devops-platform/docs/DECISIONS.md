# アーキテクチャ決定記録（ADR）

このドキュメントは、ai-devops-platformプロジェクトの重要な技術的決定とその背景を記録します。

---

## ADR-001: Z.AI GLM5 をプライマリLLMに採用

### ステータス
✅ 採用済み

### コンテキスト
自動開発プラットフォームには、コード生成・理解能力の高いLLMが必要。

### 決定
Z.AI GLM5（glm-5-nano）をプライマリモデルとして採用。フォールバックとしてGLM4.6、OpenAI GPT-4o-miniを用意。

### 理由
1. **コスト効率**: GLM5は同等性能の他モデルより低コスト
2. **日本語対応**: 日本語プロンプトでの性能が良好
3. **コード生成能力**: TypeScript/JavaScript生成が高品質
4. **API安定性**: Z.AI APIが安定稼働している

### 代替案
- **OpenAI GPT-4**: 高いがコストが高い
- **Claude 3.5 Sonnet**: 高性能だがコストが高い
- **Gemini Pro**: コスト良好だが日本語性能が劣る

### 影響
- Z.AI API Key が必要
- レート制限（60 req/min）を考慮した設計が必要

## ADR-002: GitHub Actions をオーケストレーターに採用

### ステータス
✅ 採用済み

### コンテキスト
タスクのトリガー、実行、監視を行うオーケストレーション層が必要。

### 決定
GitHub Actions をオーケストレーターとして採用。

### 理由
1. **GitHub統合**: Issue/PR/Workflow とシームレスに連携
2. **コスト**: パブリックリポジトリなら無料、プライベートも月2000分無料
3. **マトリクスビルド**: 並列実行が容易
4. **Secrets管理**: GitHub Secrets で安全に管理

### 代替案
- **自前サーバー + Cron**: メンテナンスコストが高い
- **AWS Step Functions**: コストが高い、GitHub統合が面倒
- **CircleCI**: GitHub Actions と同等だが GitHub ネイティブではない

### 影響
- GitHub Actions の制限（同時実行数、実行時間6時間制限）に準拠必要
- ワークフロー定義が YAML で管理される

## ADR-003: temperature=0.0 で決定論的出力

### ステータス
✅ 採用済み

### コンテキスト
同じタスクに対して一貫した結果を得る必要がある。

### 決定
全てのLLM呼び出しで `temperature=0.0` を使用。

### 理由
1. **再現性**: 同じ入力 → 同じ出力（ほぼ）
2. **デバッグ**: エラー時の原因特定が容易
3. **品質**: コード生成では創造性より正確性が重要

### 代替案
- **temperature=0.1-0.3**: 若干の多様性を許可
- **タスク別に可変**: 創造的タスクでは高く設定

### 影響
- 同じプロンプトで同じ結果が期待できる（完全同一ではない）
- 多様な解決策の探索には不向き

## ADR-004: Result<T, E> パターンでエラーハンドリング

### ステータス
✅ 採用済み

### コンテキスト
例外ではなく明示的なエラー処理を行いたい。

### 決定
Rust風の `Result<T, E>` 型パターンを採用。

```typescript
type Result<T, E = Error> = 
  | { success: true; data: T }
  | { success: false; error: E }
```

### 理由
1. **明示的**: 全てのエラーパスが型で表現される
2. **強制**: エラー処理を忘れるとコンパイルエラー
3. **構造化**: エラー情報を構造的に保持できる

### 代替案
- **try-catch**: エラー処理が暗黙的になりがち
- **Either<L, R>**: Result と同等だが命名が Result の方が直感的

### 影響
- 全ての非同期関数が `AsyncResult<T, E>` を返す
- エラー分類が明確になる

## ADR-005: 最大3回の自動修正試行

### ステータス
✅ 採用済み

### コンテキスト
品質ゲート失敗時に自動修正を試みるが、無限ループを避ける必要がある。

### 決定
最大3回の自動修正試行まで許可。4回目で人間に通知。

### 理由
1. **効率**: 3回で解決しない問題は根本的に難しい
2. **コスト**: LLM呼び出しコストの増加を防止
3. **経験則**: 多くの問題は1-2回で解決する

### 代替案
- **無制限**: 無限ループのリスク
- **1回のみ**: 解決率が下がる
- **5回**: コスト増、3回と大差ない

### 影響
- 3回失敗で `ai-blocked` ラベル付与
- 人間の介入が必要

## ADR-006: ハイブリッドトリガー（定期 + インスタント）

### ステータス
✅ 採用済み

### コンテキスト
タスクの即座の処理と、バックグラウンドでの継続的処理の両方が必要。

### 決定
15分ごとの定期ポーリング + `/opencode` コメントによるインスタントトリガーの併用。

### 理由
1. **即応性**: 緊急タスクを即座に処理可能
2. **効率**: アイドル時のリソース消費を抑制
3. **予測可能性**: 定期実行で確実に処理される

### 代替案
- **定期のみ**: 即座の処理ができない
- **Webhookのみ**: 設定が複雑、外部サービス依存
- **常時ポーリング（1分）**: API呼び出し回数増加

### 影響
- `/opencode` コメントで即座に処理開始
- 15分以内に必ずタスクがピックアップされる

## ADR-007: 最大2並列セッション

### ステータス
✅ 採用済み

### コンテキスト
複数タスクの同時処理が必要だが、リソース制限がある。

### 決定
最大2セッションまで並列実行を許可。

### 理由
1. **安全性**: GitHub Actions の並列制限（通常3-5）に対して余裕
2. **コスト**: 同時LLM呼び出しの制御
3. **品質**: 並列数が多いとリソース競合で品質低下

### 代替案
- **1並列**: 安全だが効率が悪い
- **3並列**: GitHub Actions の制限に近い、リスクあり
- **動的**: 複雑すぎる

### 影響
- 同時に2つのIssueまで処理可能
- 3つ目以降は待機

## ADR-008: auto-merge ラベルで明示的な自動マージ許可

### ステータス
✅ 採用済み

### コンテキスト
自動マージは危険な操作のため、慎重な制御が必要。

### 決定
`auto-merge` ラベルが明示的に付いている場合のみ自動マージを許可。

### 理由
1. **安全性**: 誤マージを防止
2. **制御**: 人間が明示的に許可したものだけマージ
3. **監査**: どのPRが自動マージされたか明確

### 代替案
- **全PR自動マージ**: 危険すぎる
- **設定ファイルで制御**: ラベルの方が柔軟
- **ブランチごと**: main への直接マージなど危険

### 影響
- `auto-merge` ラベルがないPRは自動マージされない
- 人間がラベルを付けることで明示的な許可

## ADR-009: OpenCode を実行エンジンに採用

### ステータス
✅ 採用済み

### コンテキスト
GitHub Actions 内で LLM を活用したコード生成を実行するエンジンが必要。

### 決定
OpenCode を実行エンジンとして採用。

### 理由
1. **CLI統合**: GitHub Actions から簡単に呼び出し可能
2. **マルチモデル対応**: Z.AI, OpenAI, Claude, Gemini に対応
3. **カスタマイズ**: 独自コマンド・ツールの追加が可能
4. **開発中**: 機能追加・改善が活発

### 代替案
- **GitHub Copilot**: APIアクセスが制限的
- **自前実装**: 開発コストが高い
- **他AIエージェント**: GitHub Actions 統合が難しい

### 影響
- OpenCode CLI が GitHub Actions ランナーに必要
- カスタムツールを `.opencode/` に配置

## ADR-010: アイドルモードでバックグラウンドタスク実行

### ステータス
🟡 検討中

### コンテキスト
処理すべきタスクがない時間を有効活用したい。

### 決定
アイドルモードで以下のタスクを実行：
1. テストカバレッジ向上
2. セキュリティ監査
3. ドキュメント更新
4. 依存関係更新確認

### 理由
1. **効率**: リソースを無駄にしない
2. **継続的改善**: コードベースが徐々に改善される
3. **低リスク**: 緊急でないタスクなので失敗しても影響小

### 代替案
- **アイドル時は何もしない**: リソース活用なし
- **夜間バッチ**: 時間指定が面倒

### 影響
- 常にコードベースが改善される
- アイドルモードタスク用のラベル・設定が必要
