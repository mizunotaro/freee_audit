# 開発ロードマップ

## フェーズ概要

```
Phase 0: 基盤構築（現在）
    ↓
Phase 1: MVP検証
    ↓
Phase 2: 本格運用
    ↓
Phase 3: 拡張・最適化
    ↓
Phase 4: エコシステム展開
```

---

## Phase 0: 基盤構築（1-2週間）

### 目標
中央ハブ方式の最小限の動作確認可能な状態を作る

### タスク

#### Week 1: 環境セットアップ
- [ ] **P0-001**: ai-devops-platform リポジトリ作成
- [ ] **P0-002**: ファイル一式を移行・コミット
- [ ] **P0-003**: GitHub Secrets 設定
  - `ZAI_API_KEY`
  - `HUB_PAT` (全ターゲットリポジトリへの書き込み権限)
- [ ] **P0-004**: GitHub Variables 設定
  - `AI_AUTOMATION_MODE=enabled`
- [ ] **P0-005**: ラベル作成（中央ハブ + ターゲット）
  - 中央ハブ用: `ai-task`, `ai-in-progress`, `ai-completed`, `ai-blocked`
  - ターゲット用: 同上（各ターゲットリポジトリに作成）

#### Week 2: 動作確認
- [ ] **P0-006**: freee_audit にテスト用Issue作成
  - `ai-task` ラベル付きで作成
  - シンプルなタスクで確認（例: "Add hello() function"）
- [ ] **P0-007**: 中央ハブのワークフローがスキャンすることを確認
  - 15分ポーリングの動作確認
  - ログでタスク検出を確認
- [ ] **P0-008**: タスクが処理されPRが作成されることを確認
  - 品質ゲート実行確認
  - PRがターゲットリポジトリに作成されることを確認

### 完了条件
- [ ] テストIssueが自動的に処理され、PRが作成される
- [ ] 品質ゲートが全てPASSする
- [ ] 中央ハブからターゲットリポジトリへのPR作成が成功する

- [ ] **P1-006**: 型エラー時の自動修正確認

#### Week 5: 運用データ収集
- [ ] **P1-008**: セッション実行時間の統計収集
- [ ] **P1-009**: 品質ゲート通過率の測定
- [ ] **P1-010**: 自動修正成功率の測定
- [ ] **P1-011**: LLM APIコストの測定

### 完了条件
- [ ] 10個以上の実用的タスクが自動処理される
- [ ] 品質ゲート通過率 > 80%
- [ ] 自動修正成功率 > 50%

## リスク管理

### 高リスク

| リスク | 影響 | 対策 | 発生確率 |
|--------|------|------|---------|
| LLM API コスト超過 | 予算オーバー | 月次予算アラート、タスク優先度制御 | 中 |
| 品質ゲート通過率低下 | 自動化失敗 | プロンプト改善、タスク分割 | 中 |
| セキュリティインシデント | 信頼損失 | 入力検証強化、監査ログ | 低 |

### 中リスク

| リスク | 影響 | 対策 | 発生確率 |
|--------|------|------|---------|
| GitHub API レート制限 | 処理遅延 | キャッシング、バックオフ | 高 |
| LLM出力の不安定性 | 品質バラつき | 出力バリデーション、再試行 | 中 |
| 依存関係の複雑化 | 処理順序問題 | 依存グラフ可視化 | 中 |
