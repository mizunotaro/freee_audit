## 3. 品質ゲート（必須）

全ての実装で以下を**必ず**パスすること：

### 3.1 必須チェック

```bash
# 1. 型チェック（エラー0件）
pnpm tsc --noEmit

# 2. リント（警告0件）
pnpm eslint src/ --max-warnings=0

# 3. テスト（全PASS）
pnpm jest --passWithNoTests --coverage

# 4. ビルド（成功）
pnpm build
```

### 3.2 カバレッジ要件

| レベル | 値 | アクション |
|--------|---|-----------|
| 目標 | 100% | アイドル時自動改善 |
| 最低 | 80% | 下回る場合は `ai:blocked` |

### 3.3 品質ゲートフロー

```
実行完了
    │
    ▼
┌─────────────────┐
│ 型チェック      │──FAIL──▶ 自動修復ループ
└────────┬────────┘
         │ PASS
         ▼
┌─────────────────┐
│ リント          │──FAIL──▶ 自動修復ループ
└────────┬────────┘
         │ PASS
         ▼
┌─────────────────┐
│ テスト          │──FAIL──▶ 自動修復ループ
└────────┬────────┘
         │ PASS
         ▼
┌─────────────────┐
│ ビルド          │──FAIL──▶ 自動修復ループ
└────────┬────────┘
         │ PASS

    自動マージ判定
```

## 4. 自動マージポリシー

### 4.1 自動マージ条件（全て満たすこと）

- [x] 型チェック成功（エラー0件）
- [x] リント成功（警告0件）
- [x] 全テストPASS
- [x] ビルド成功
- [x] カバレッジ >= 80%
- [x] セキュリティスキャンクリア
- [x] Breaking Changeなし

### 4.2 自動マージ除外（人間レビュー必須）

| カテゴリ | 対象 |
|----------|------|
| セキュリティ | 認証・認可、暗号化、.env変更 |
| インフラ | DB スキーマ、環境変数、CI/CD設定 |
| 破壊的変更 | API 変更、メジャーアップデート |

### 4.3 マージ方式

- **方式**: Squash Merge
- **ブランチ削除**: 自動

## 5. 自動修復ポリシー

### 5.1 修復ループ

```
品質ゲート失敗
      │
      ▼
┌─────────────────┐
│ エラー分析      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 自動修正        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 品質ゲート再実行│──FAIL──┐
└────────┬────────┘        │
         │ PASS            │
         ▼                 │
    自動マージへ           │
                           │
         ┌─────────────────┘
         │
         ▼
   試行回数 < 3?
      │ YES ──▶ 修復ループ継続
      │ NO
      ▼
   ai:blocked へ遷移
   人間に通知
```

### 5.2 修復対象

| エラー種別 | 自動修復 | 最大試行 |
|-----------|----------|----------|
| 型エラー | 可能 | 3回 |
| リントエラー | 可能 | 3回 |
| テスト失敗 | 可能 | 3回 |
| ビルドエラー | 可能 | 3回 |
| セキュリティ問題 | 不可 | 0回 |

## 6. レートリミット管理

### 6.1 Z.AI GLM5 制限

| 制限 | 公称値 | 安全値 | 理由 |
|------|--------|--------|------|
| Concurrent Sessions | 3 | **2** | 安全マージン |
| Time Window | 5時間 | - | リセット待ち |

### 6.2 制御方法

```yaml
# 実行前チェック
1. 過去1時間の実行中ジョブ数を取得
2. 実行中 < 2 の場合のみ実行許可
3. 実行中 >= 2 の場合、待機してリトライ
```

### 6.3 バックオフ戦略

| 試行回数 | 待機時間 |
|----------|----------|
| 1回目 | 60秒 |
| 2回目 | 120秒 |
| 3回目 | 240秒 |
| 最大 | 300秒 |

## 7. セッション管理

### 7.1 異常検知条件

| 条件 | 閾値 | 対処 |
|------|------|------|
| レスポンスなし | 30分 | `ai:ready` に戻す |
| 同一エラー繰り返し | 3回 | `ai:blocked` に遷移 |
| ワークフロー実行時間 | 1時間超過 | 強制キャンセル |

### 7.2 健全性監視

```yaml
# 10分ごとに実行
health-monitor.yml:
  - 30分以上更新のない ai:in-progress Issue を検出
  - 1時間以上実行中のワークフローをキャンセル
  - 孤立したラベル（PRなしで ai:pr-open）を修正
```

## 9. 禁止事項

### 9.1 絶対禁止

| 禁止事項 | 理由 |

| 操作 | 条件 |
|------|------|

## 12. OpenCode カスタムコマンド

### 12.1 /issue コマンド

```bash
# 通常作成
/issue 新機能: ログイン機能

# 即時実行
/issue バグ修正 --immediate --priority=p1 --type=bug

# 別リポジトリ
/issue 新機能 --repo=owner/repo
```

### 12.2 オプション

| オプション | 短縮 | 説明 | デフォルト |
|-----------|------|------|-----------|
| `--immediate` | `-i` | 即時実行 | false |
| `--priority=p1\|p2\|p3` | | 優先度 | p2 |
| `--type=feature\|bug\|...` | | タイプ | feature |
| `--repo=owner/repo` | | 対象リポジトリ | 現在 |

## 13. フォールバックロジック

### 13.1 エラー分類

| 分類 | 対処 | 例 |
|------|------|-----|
| Retryable | 自動リトライ | レートリミット、タイムアウト |
| Recoverable | 代替手段 | モデルエラー、セッションハング |
| Fatal | 人間確認 | 認証エラー、予算超過 |

### 13.2 復旧アクション

| エラー | アクション |
|--------|-----------|
| レートリミット | 指数バックオフで待機 |
| モデルエラー | フォールバックモデルへ切替 |
| セッションハング | 新セッション起動 |
| 品質ゲート失敗 | 自動修復ループ |
| 依存関係エラー | キャッシュクリア・再試行 |
